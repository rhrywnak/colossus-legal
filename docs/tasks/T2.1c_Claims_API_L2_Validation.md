# Task: T2.1c – Claims API L2 (Validation & Error Handling)

title: "T2.1c – Claims API L2 (Validation & Error Handling)"
instructions:
  This task adds validation and proper error handling to the Claims API.

  The goal is:
    - Reject invalid inputs for create/update operations.
    - Return consistent, structured error responses.
    - Add tests that cover both success and error cases.

context:
  repo: colossus-legal
  persona: BackendAgent
  backend-root: ./backend
  docs:
    - AGENTS.md
    - docs/WORKFLOW.md
    - docs/DEV_ONBOARDING.md
    - docs/TASK_TRACKER_V2.md
    - docs/API_DESIGN.md

requirements:
  - Review the Claims create/update endpoints in `backend/src/api/claims.rs` (or equivalent).
  - Introduce validation logic (e.g., non-empty title, allowed status values).
  - Use a consistent error response format (e.g., `{ "error": "...", "message": "...", "details": {...} }`).
  - Map different error conditions to appropriate HTTP status codes (400, 404, 500).
  - Run:
      - `cargo check --manifest-path backend/Cargo.toml`
      - `cargo test --tests --manifest-path backend/Cargo.toml`

tests:
  - Required.
  - Add or extend tests (e.g., in `backend/tests/claims_validation.rs`):
      - Test invalid payloads return 400 with the expected error body.
      - Test non-existent IDs return 404 where appropriate.
      - Test valid payloads still succeed as expected.

acceptance_criteria:
  - Invalid input is rejected with clear, structured error responses.
  - Tests cover both success and failure conditions.
  - No regressions to earlier happy-path behavior from T2.1b.
  - TASK_TRACKER_V2 updated for T2.1c status.

files:
  - backend/src/api/claims.rs
  - backend/src/error.rs (if you centralize error types)
  - backend/tests/claims_validation.rs
  - docs/TASK_TRACKER.md
