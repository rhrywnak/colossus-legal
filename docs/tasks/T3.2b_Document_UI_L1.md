# T3.2b — Document UI L1 (Real API Integration + Basic Tests)

## Task ID
T3.2b

## Layer
L1 (real data, happy path + basic tests)

## Persona
FrontendAgent

## Branch
`feature/T3.2b-document-ui-l1`

## Goal

Wire the Documents UI to the **real backend `/documents` API** and add at least one Vitest test covering the
service and/or component behavior, similar to Claims UI L1.

## Prerequisites

- T3.2a completed:
  - `/documents` route exists.
  - `DocumentsPage.tsx` exists with stubbed logic.
  - `getDocumentsStub()` implemented.

- Backend T3.1b completed:
  - Real `GET /documents` implemented.

- `VITE_API_URL` is configured in the frontend environment.

## Requirements

### 1. Service

- Extend or replace `getDocumentsStub()` with a real API call under `frontend/src/services/documents.ts`, e.g.:

  ```ts
  const API_BASE_URL = import.meta.env.VITE_API_URL;

  export async function getDocuments(): Promise<DocumentItem[]> {
    const res = await fetch(`${API_BASE_URL}/documents`);
    if (!res.ok) {
        throw new Error("Failed to fetch documents");
    }
    const data = await res.json();
    // Optionally validate/normalize shape here
    return data;
  }
  ```

- You may keep `getDocumentsStub()` as a helper if it is useful for tests, but the page should use `getDocuments()`.

### 2. DocumentsPage

- Update `DocumentsPage.tsx` to call `getDocuments()` instead of `getDocumentsStub()`.

- Ensure the four states are handled:
  - Loading
  - Empty
  - Success
  - Error

### 3. Tests

- Add or extend a Vitest test file, e.g. `frontend/src/services/__tests__/documents.test.ts`:

  - Mock `fetch` or your HTTP helper.
  - Cover at least:
    - Success (non-empty list).
    - Empty list.
    - Error (non-OK response or network error).

- Optionally add a component test for `DocumentsPage` with React Testing Library, mocking the service.

### 4. Commands

- Run:

  ```bash
  npm run test
  npm run build
  ```

## Out of Scope

- Advanced filtering/sorting.
- Analysis UI.
- Any new routes.

## Acceptance Criteria

- `/documents` shows real data from the backend (when available).
- The UI behaves correctly across loading/empty/success/error states.
- At least one Vitest test for the Documents service and/or page passes.
- `npm run test` and `npm run build` both pass.
