# T3.2a — Document UI L0 (Skeleton Page + Stub Service)

## Task ID
T3.2a

## Layer
L0 (UI skeleton, stubbed data)

## Persona
FrontendAgent

## Branch
`feature/T3.2a-document-ui-l0`

## Goal

Create a **Documents UI skeleton** that provides a `/documents` page with basic loading/empty/error/success states,
hooked up to a stub service (`getDocumentsStub()`), similar to Claims L0.

## Prerequisites

- Backend T3.1a may or may not be complete; this task does not depend on real Document API data.
- Frontend project compiles and runs (`npm run dev`).

## Requirements

### 1. Route

- Add a route `/documents` to the frontend router (`App.tsx` or your routing setup).
- Ensure there is a navigation path or direct URL access to `/documents`.

### 2. Page Component

- Create `frontend/src/pages/DocumentsPage.tsx`:

  - Uses `useState` + `useEffect` (or React Query if already standardised) to call a stub service.
  - Renders at least four UI states:
    - **Loading** (e.g. "Loading documents…").
    - **Error** (e.g. "Failed to load documents").
    - **Empty** (e.g. "No documents found").
    - **Success** (renders a list of stubbed documents, e.g. title + doc_type).

### 3. Stub Service

- Create `frontend/src/services/documents.ts` with:

  ```ts
  export type DocumentItem = {
    id: string;
    title: string;
    docType: string;
    createdAt?: string;
  };

  export async function getDocumentsStub(): Promise<DocumentItem[]> {
    // hard-coded array of documents
  }
  ```

- For L0, the data can be hard-coded in-memory.

### 4. Wiring

- `DocumentsPage` should import and use `getDocumentsStub()` at this layer.
- No real HTTP calls yet.

### 5. Compile & Run

- `npm run build` should succeed.
- `npm run dev` and visiting `/documents` should show the stub-based UI without runtime errors.

## Tests

- Optional at L0.
- You may add a basic test for the stub service or component if desired, but it is not required.

## Acceptance Criteria

- `/documents` renders a page that transitions through loading → success (with stub data) or error/empty states when simulated.
- No TypeScript errors or runtime exceptions for this route.
- Frontend build (`npm run build`) passes.
