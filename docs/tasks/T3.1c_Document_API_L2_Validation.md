# T3.1c — Document API L2 (Validation & Error Handling)

## Task ID
T3.1c

## Layer
L2 (validation, 400/404, structured errors)

## Persona
BackendAgent

## Branch
`feature/T3.1c-document-api-l2-validation`

## Goal

Extend the Document API to support **create/read/update by ID** with proper input validation and structured error
handling, mirroring Claims L2 behavior.

## Prerequisites

- T3.1a and T3.1b completed and merged:
  - `GET /documents` wired to Neo4j.
  - `Document` model and `DocumentDto` in place.

- Structured error type already exists (from Claims) in `backend/src/error.rs` and is exported via `lib.rs`.

## Requirements

### 1. Endpoints

Implement the following endpoints in `backend/src/api/documents.rs` and register them in `api/mod.rs`:

- `POST /documents`
  - Creates a new Document metadata entry.
- `GET /documents/{id}`
  - Retrieves Document by ID.
- `PUT /documents/{id}`
  - Updates Document metadata.

### 2. DTOs

Add or refine request DTOs in `backend/src/dto/document.rs`:

- `DocumentCreateRequest`:
  - `title: String`
  - `doc_type: String`
  - `created_at: Option<String>`

- `DocumentUpdateRequest`:
  - Optional fields for partial update, e.g.:
    - `title: Option<String>`
    - `doc_type: Option<String>`
    - `created_at: Option<String>`

### 3. Validation Rules

- `title` must be non-empty (on create; if provided on update).
- `doc_type` must be one of an allowed set (e.g. `pdf`, `motion`, `ruling`, `evidence`, `filing`).
- If present, `created_at` must be a valid ISO-8601 timestamp.
- If validation fails, return 400 with structured error.

### 4. Error Handling

Use the existing structured error shape:

```json
{ "error": "...", "message": "...", "details": { ... } }
```

- 400 for validation failures.
- 404 for non-existent IDs on GET/PUT.
- 500-style for unexpected internal errors (e.g., repository/Neo4j failures).

### 5. Repository

Extend `DocumentRepository` with:

- `create_document(request: DocumentCreateRequest) -> Result<Document, RepositoryError>`
- `get_document_by_id(id: &str) -> Result<Document, RepositoryError>`
- `update_document(id: &str, request: DocumentUpdateRequest) -> Result<Document, RepositoryError>`

`RepositoryError` variants should support mapping to 404 and 500 outcomes.

### 6. Integration Tests

Add `backend/tests/documents_validation.rs`:

- **400 cases**:
  - POST with empty `title`.
  - POST with invalid `doc_type`.
  - POST with invalid `created_at`.
  - PUT with invalid fields.

- **404 cases**:
  - GET /documents/{id} with a non-existent ID.
  - PUT /documents/{id} with a non-existent ID.

- **Happy-path**:
  - POST → GET → PUT roundtrip with valid data.

Use the same test marker pattern as Claims tests (`source: "test"`, `test_run_id`) and clean up only test data.

### 7. Commands

Run:

```bash
cargo check --manifest-path backend/Cargo.toml
cargo test --tests --manifest-path backend/Cargo.toml
```

## Out of Scope

- Analysis endpoints (T3.1d).
- Any Document relationships to Claims/Evidence (those are for L3 and later phases).
- Frontend changes.

## Acceptance Criteria

- Document create/read/update endpoints behave correctly and return structured errors for bad inputs.
- `backend/tests/documents_validation.rs` passes and covers:
  - 400 invalid payloads.
  - 404 unknown IDs.
  - Happy path.
- `cargo test --tests` remains green, including existing tests.
