# Task: T1.5 – Backend Dev Env Configuration (Runtime Readiness & Test Isolation)

title: "T1.5 – Backend Dev Env Configuration (Runtime Readiness & Test Isolation)"
instructions:
  This task ensures that:
    - `cargo run` works reliably in dev without ad-hoc env var exports.
    - Claims integration tests do NOT clear all Claim nodes, but only their own test data.

  It has two parts:
    1) Implement dotenv-based backend env loading.
    2) Adjust Claims integration tests to use a `test_run_id` (or similar) and delete only test data.

context:
  repo: colossus-legal
  persona: BackendAgent + DocsAgent
  backend-root: ./backend
  docs:
    - AGENTS.md
    - docs/WORKFLOW.md
    - docs/DEV_ONBOARDING.md
    - docs/TASK_TRACKER.md

requirements:
  # 1) dotenv-based env loading

  - Ensure `dotenvy` is included as a dependency in `backend/Cargo.toml` (if not already present).
  - In `backend/src/main.rs`, call `dotenvy::dotenv().ok();` near the top of `main` (or early in startup)
    so that `backend/.env` is loaded automatically in dev.
  - Ensure:
      - `backend/.env.example` exists with placeholders for:
          - NEO4J_URI
          - NEO4J_USER
          - NEO4J_PASSWORD
          - BACKEND_PORT
      - `backend/.env` is listed in `.gitignore` so real secrets are not committed.
  - After this, with a properly configured `backend/.env`, `cargo run --manifest-path backend/Cargo.toml`
    MUST start without env-var panics.

  # 2) Test isolation for Claims integration tests

  - Update `backend/tests/claims_list.rs` so that:
      - When inserting Claim nodes for tests, they include a unique marker:
          - e.g. a `test_run_id` property or a specific `source: "test"` field.
      - Deletion/cleanup removes only nodes with that marker, NOT all `:Claim` nodes:
          - e.g. `MATCH (c:Claim {source: "test"}) DETACH DELETE c`.
      - Both "non-empty" and "empty" tests:
          - insert and clean up their own data safely.
  - The tests must still:
      - Verify the non-empty case (at least one Claim).
      - Verify the empty case.
      - Use real Neo4j.

  - Run:
      - `cargo check --manifest-path backend/Cargo.toml`
      - `cargo test --tests --manifest-path backend/Cargo.toml`
      - `cargo run --manifest-path backend/Cargo.toml` (with backend/.env configured)

  # 3) Documentation

  - Update `docs/DEV_ONBOARDING.md`:
      - Add/ensure a clear "Backend dev env setup" section:
          - Copy `backend/.env.example` → `backend/.env`.
          - Fill in NEO4J_URI/USER/PASSWORD and BACKEND_PORT.
          - Ensure Neo4j is running.
          - Run `cargo run --manifest-path backend/Cargo.toml` and check `/health` + `/api/status`.
  - Update `docs/TASK_TRACKER.md`:
      - Mark T1.5 as DONE with a short note:
          - dotenv dev env loading implemented.
          - Claim tests use a test marker and no longer delete all Claim nodes.

acceptance_criteria:
  - With a correctly configured `backend/.env`, `cargo run` starts without env-var panics.
  - Claims integration tests use a test marker (e.g. `source: "test"` or `test_run_id`) and only delete
    marked nodes; other Claim data is untouched.
  - `cargo test --tests` passes.
  - `docs/DEV_ONBOARDING.md` and `docs/TASK_TRACKER.md` fully reflect the new dev env and test isolation behavior.
