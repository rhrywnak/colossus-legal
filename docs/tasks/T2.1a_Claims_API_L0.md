# Task: T2.1a – Claims API L0 (Skeleton Routes + Stubs, Compile-Only)

title: "T2.1a – Claims API L0 (Skeleton Routes + Stubs, Compile-Only)"
instructions:
  This task implements **Layer 0** for the Claims API.

  The goal for L0 is:
    - Have a compiling `GET /claims` route and handler in the backend.
    - Return a stubbed `ClaimDto` list (or empty array) from that handler.
    - Introduce NO Neo4j logic in this layer.

  **Important:** Because the current backend startup path requires Neo4j env vars,
  this task's acceptance criteria are intentionally limited to **compile-level**.
  Runtime configuration (env vars, dotenv, etc.) for `cargo run` will be handled
  in a separate dev-env task (T1.5).

requirements:
  - Create `backend/src/dto/claim.rs` if it does not exist and define `ClaimDto`:
      - Fields: `id: String`, `title: String`, `description: Option<String>`, `status: String`.
  - Ensure `backend/src/dto/mod.rs` exports the `claim` module.
  - Create `backend/src/api/claims.rs`:
      - Implement `list_claims(State<AppState>) -> Result<Json<Vec<ClaimDto>>, StatusCode>`.
      - Return a small hard-coded list of ClaimDto (or an empty Vec) for now.
      - Do NOT use Neo4j or repositories yet.
  - Update `backend/src/api/mod.rs`:
      - Declare `mod claims;` and import the handler.
      - Wire `GET /claims` to `claims::list_claims`.
      - Keep existing `/health` and `/api/status` behavior unchanged.
  - Run:
      - `cargo check --manifest-path backend/Cargo.toml`
  - Do NOT change env var handling, Neo4j setup, or runtime configuration in this task.

acceptance_criteria:
  - Backend compiles (`cargo check`) with the new DTO, handler, and route.
  - `GET /claims` is implemented and wired in the router.
  - No Neo4j or repository logic added in this layer.
  - No changes to `/health` or `/api/status` semantics.
  - `docs/TASK_TRACKER.md` updated to reflect T2.1a status (e.g., DONE (compile-level)).
