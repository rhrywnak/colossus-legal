# T3.1b — Document API L1 (Real Neo4j List, Happy Path)

## Task ID
T3.1b

## Layer
L1 (real data, happy path)

## Persona
BackendAgent

## Branch
`feature/T3.1b-document-api-l1`

## Goal

Make `GET /documents` return **real data from Neo4j** using a `DocumentRepository::list_documents()` method.
Keep the behavior simple and happy-path focused, similar to T2.1b for Claims.

## Prerequisites

- T3.1a is complete and merged:
  - `DocumentDto` exists.
  - `Document` model exists.
  - Stubbed `GET /documents` exists and compiles.

- Neo4j dev/test instance running and accessible via `backend/.env`.

## Requirements

### 1. Repository

- Add a `DocumentRepository` implementation in `backend/src/repositories/document_repository.rs`
  (or extend an existing repository if it already exists).

- Implement a method like:

  ```rust
  pub async fn list_documents(&self) -> Result<Vec<Document>, RepositoryError> {
      // MATCH (d:Document) RETURN d ORDER BY d.created_at DESC
  }
  ```

- Use Neo4j driver already present in the project (reusing patterns from Claims repository).

### 2. API Handler

- Update `backend/src/api/documents.rs`:

  - Replace the stubbed data in `list_documents` with a call to the repository.
  - Map `Document` → `DocumentDto`.

- Keep the handler happy-path oriented:
  - If the repository call succeeds, return 200 + JSON list.
  - 500-style error handling can be minimal at this layer (more robust handling comes in L2).

### 3. Integration Tests

- Add `backend/tests/documents_list.rs` with integration tests that use the dev/test Neo4j:

  - **Non-empty case**:
    - Insert one or more `:Document` nodes with test markers (e.g. `source: "test"`, `test_run_id`).
    - Call `GET /documents`.
    - Assert:
      - 200 OK
      - Non-empty JSON array
      - At least one entry matches inserted data.

  - **Empty case**:
    - Ensure no test-marked Documents exist.
    - Call `GET /documents`.
    - Assert:
      - 200 OK
      - Empty JSON array.

- Clean up only test-marked nodes, following the same pattern used in Claims tests.

### 4. Commands

- Run:

  ```bash
  cargo check --manifest-path backend/Cargo.toml
  cargo test --tests --manifest-path backend/Cargo.toml
  ```

## Out of Scope

- Validation / 400 / 404 handling (that belongs to T3.1c).
- Analysis endpoints (T3.1d).
- Any frontend changes.

## Acceptance Criteria

- `GET /documents` returns real Neo4j-backed data.
- `backend/tests/documents_list.rs` exists and passes:
  - Non-empty list case.
  - Empty list case.
- No regressions to existing endpoints (Claims and others).
- `cargo test --tests` passes against dev/test Neo4j.
