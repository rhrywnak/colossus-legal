# T3.1d — Document API L3 (Analysis / Graph Queries)

## Task ID
T3.1d

## Layer
L3 (analysis, graph traversal, derived data)

## Persona
BackendAgent

## Branch
`feature/T3.1d-document-api-l3-analysis`

## Goal

Add a **read-only analysis endpoint** for Documents that exposes graph-derived insights, similar in spirit to
`GET /claims/analysis/refuted`, but focused on document–claim–evidence relationships.

Initial target endpoint:

- `GET /documents/analysis/claim-links`

## Prerequisites

- T3.1a–c completed and merged.
- DATA_MODEL.md defines:
  - `(:Document)-[:MENTIONS]->(:Claim)`
  - `(:Evidence)-[:APPEARS_IN]->(:Document)`
  - `(:Evidence)-[:SUPPORTS|REFUTES]->(:Claim)`

## Requirements

### 1. DTOs / Models

Add analysis DTOs in `backend/src/dto/document_analysis.rs` (or in `dto/document.rs` if you prefer to keep
related DTOs together):

Example:

```rust
pub struct DocumentClaimLinkDto {
    pub id: String,
    pub title: String,
    pub claims: Vec<DocumentClaimLinkEntry>,
}

pub struct DocumentClaimLinkEntry {
    pub id: String,
    pub title: String,
    pub link_type: String, // e.g. "MENTIONS", "SUPPORTS", "REFUTES"
}

pub struct DocumentsClaimLinksResponse {
    pub documents: Vec<DocumentClaimLinkDto>,
}
```

Export via `backend/src/dto/mod.rs`.

### 2. Repository

Add a read-only method in `DocumentRepository`, e.g.:

```rust
pub async fn list_documents_with_claim_links(&self) -> Result<Vec<DocumentClaimLink>, RepositoryError> {
    // Cypher here
}
```

Cypher sketch:

```cypher
MATCH (d:Document)
OPTIONAL MATCH (d)-[:MENTIONS]->(c1:Claim)
OPTIONAL MATCH (e:Evidence)-[:APPEARS_IN]->(d)
OPTIONAL MATCH (e)-[r2:SUPPORTS|REFUTES]->(c2:Claim)
WITH d,
     collect(DISTINCT { id: c1.id, title: c1.title, link_type: "MENTIONS" }) +
     collect(DISTINCT { id: c2.id, title: c2.title, link_type: type(r2) }) AS claim_links
RETURN d, claim_links
```

Map this into internal structs, then into DTOs.

### 3. API Handler

In `backend/src/api/documents.rs`:

- Add handler:

  ```rust
  pub async fn get_documents_claim_links(...) -> impl IntoResponse
  ```

- Use the repository method.
- On success, return 200 with `DocumentsClaimLinksResponse`.
- On repository/Neo4j failure, return 500-style structured error using the existing error type
  (`{ error, message, details }`).

### 4. Routing

In `backend/src/api/mod.rs`:

- Register:

  - `GET /documents/analysis/claim-links` → `documents::get_documents_claim_links`.

### 5. Integration Tests

Create `backend/tests/documents_analysis.rs`:

- Use dev/test Neo4j with markers (`source: "test"`, `test_run_id`).
- Test scenarios:

  1. **Documents with claim links**:
     - Create:
       - A Document.
       - A Claim.
       - Optionally Evidence connecting them.
     - Create relationships:
       - `(:Document)-[:MENTIONS]->(:Claim)`
       - Optionally `(:Evidence)-[:APPEARS_IN]->(:Document)` + `(:Evidence)-[:SUPPORTS|REFUTES]->(:Claim)`
     - Call `GET /documents/analysis/claim-links`.
     - Assert:
       - 200 OK.
       - The Document appears with appropriate claim link entries.

  2. **Empty graph / no links**:
     - Ensure no test-marked Documents/Evidence/Claims exist.
     - Call the endpoint.
     - Assert:
       - 200 OK.
       - `documents` array is empty.

- Clean up only test-marked nodes/relationships.

Run:

```bash
cargo check --manifest-path backend/Cargo.toml
cargo test --tests --manifest-path backend/Cargo.toml
```
